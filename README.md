# Прототип детекции уборки столиков по видео

Упрощённый пайплайн на одном видео: **YOLOv8n** (класс «человек»), **один столик** в зоне ROI, события **пусто / занят / подход после пустоты**, таблица событий в **Pandas**, минимальная **визуализация** (цвет ROI) и **средняя задержка** между «стол опустел» и «следующий подошёл».

## Как запустить

### Вариант A: uv (рекомендуется)

```bash
uv sync --group dev
uv run python main.py --video "videos/видео 1.mp4"
```

Окружение создаётся в `.venv` автоматически (`uv` управляет зависимостями и lock-файлом).

### Вариант B: pip

```bash
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python main.py --video "videos/видео 1.mp4"
```

При первом запуске Ultralytics скачает веса **`yolov8n.pt`** в рабочую директорию.

### Логирование (Loguru)

#### Куда пишутся логи

- Каталог **`logs/`** в корне проекта создаётся при запуске, если его ещё нет.
- Файлы: **`logs/detection_YYYY-MM-DD.log`** (один файл на календарный день).
- **Ротация:** новый файл начинается в **00:00** каждого дня.
- **Хранение:** файлы старше **7 дней** удаляются автоматически.
- Настройки в **`.env`** в корне проекта; шаблон — **`.env.example`** (скопируйте и переименуйте).

#### Как устроен порог уровня

В Loguru задаётся **минимальный уровень сообщений**, которые вы хотите видеть. В лог попадают сообщения **не слабее** этого порога (по важности встроенной шкалы Loguru: от `TRACE` до `CRITICAL`). Сообщения «тише» порога (например, `DEBUG` при пороге `INFO`) **не выводятся**.

Проще говоря: **больше цифра в настройке — больше типов сообщений в логе** (подробнее вывод). **Меньше цифра — тише и короче лог** (только важное).

#### Шкала цифр `1`–`7`

В проекте используется **семь уровней** — цифровая шкала для удобства. Внутри она переводится в имена уровней Loguru. **Цифра 7** — максимум: **все** типы сообщений (порог `TRACE`).

| Цифра | Имя Loguru | Что обычно попадает в лог | Когда использовать |
|-------|------------|---------------------------|---------------------|
| **1** | `CRITICAL` | Только аварийные сообщения (`CRITICAL`) | Почти всегда тихо; только если нужно видеть только «падения» |
| **2** | `ERROR` | `ERROR` и `CRITICAL` | Ошибки выполнения и критичные сбои |
| **3** | `WARNING` | `WARNING` … `CRITICAL` | Предупреждения, ошибки и критичные события |
| **4** | `INFO` | `INFO` … `CRITICAL` | **Режим по умолчанию:** нормальный ход работы, события, отчёты |
| **5** | `SUCCESS` | `SUCCESS`, `WARNING`, `ERROR`, `CRITICAL` | Строже, чем **4**: обычные сообщения `INFO` в лог **не** попадают (нужен, если хотите отфильтровать «шум» `INFO`) |
| **6** | `DEBUG` | `DEBUG` … `CRITICAL` | Отладка: дополнительные технические сообщения |
| **7** | `TRACE` | Все уровни, включая самые мелкие | **Максимальная детализация:** полный вывод (в Loguru это нижний порог) |

Вместо цифр можно указать **имя** уровня (`INFO`, `DEBUG`, `WARNING`, …) — поведение совпадает с соответствующим порогом в Loguru.

#### Переменные в `.env`

| Переменная | Назначение |
|------------|------------|
| `LOG_LEVEL` | Базовый уровень **и для консоли, и для файла**, если не переопределено ниже. По умолчанию **4** (`INFO`). |
| `LOG_CONSOLE_LEVEL` | Только **вывод в терминал** (stderr). Если строка **пустая** или переменная не задана — используется `LOG_LEVEL`. |
| `LOG_FILE_LEVEL` | Только **запись в файлы** в `logs/`. Если пусто — снова `LOG_LEVEL`. |

Так можно развести режимы: например, в консоли смотреть всё подряд (**7**), а на диск писать только предупреждения и выше (**3**), чтобы не раздувать файлы.

#### Примеры

Один уровень везде (по умолчанию, нормальный запуск):

```env
LOG_LEVEL=4
```

Максимум подробностей везде (все уровни):

```env
LOG_LEVEL=7
```

То же через имя:

```env
LOG_LEVEL=TRACE
```

Раздельно консоль и файл:

```env
LOG_LEVEL=4
LOG_CONSOLE_LEVEL=7
LOG_FILE_LEVEL=3
```

Здесь: в терминале **максимум** (7), в `logs/` — от **WARNING** и выше (3), если не заданы другие переменные.

### Как выбрать ROI и что происходит дальше

**Что такое ROI.** Это **один прямоугольник** на **первом кадре** видео, внутри которого лежит **один** столик, который вы хотите отслеживать. Все события (пусто / занято / подход) считаются только в этой зоне.

**Пошагово:**

1. Запускаете команду с `--video` (без `--skip-roi`).
2. Открывается **окно с первым кадром** ролика (интерфейс OpenCV).
3. **Мышью** нажимаете и тянете **прямоугольник** вокруг нужного столика (верхний левый угол → нижний правый).
4. Подтверждаете выбор: **Enter** или **Пробел** (как пишет подсказка OpenCV в консоли: `SPACE or ENTER`).
5. Отмена выбора — клавиша **C** (вернётся пустой ROI и будет ошибка).
6. После подтверждения в лог попадут числа `(x, y, w, h)` — это координаты зоны в пикселях.

**После ROI программа не завершилась** — дальше идёт так:

| Этап | Что вы видите | Важно |
|------|----------------|--------|
| Загрузка **YOLO** | В терминале может идти строка `Downloading ... yolov8n.pt` с процентами | **Первый запуск:** файл ~6 МБ качается с GitHub. **Не нажимайте Ctrl+C** — дождитесь 100%. При следующих запусках файл уже есть в папке проекта, загрузка будет сразу. |
| Обработка видео | Долго может **ничего не выводиться** в консоль (или редко — каждые 500 кадров на уровне DEBUG) | Идёт прогон **всех кадров** с YOLO. Ролик 2560×1440 и длинный по времени обрабатывается **медленно** — это нормально. |
| Конец | В логе `Пайплайн завершён`, файлы в **`outputs/`** | `outputs/output.mp4`, `outputs/report.txt` |

**Заранее скачать веса** (чтобы не ждать при первом запуске с ROI):

```bash
uv run python -c "from ultralytics import YOLO; YOLO('yolov8n.pt')"
```

или положите вручную **`yolov8n.pt`** в корень проекта.

**Без окна выбора** (вся картинка = зона столика, для проверки пайплайна без GUI):

```bash
uv run python main.py --video "videos/видео 1.mp4" --skip-roi
```

### Если окно ROI не открывается (Wayland / Qt / `NULL window handler`)

На GNOME + Wayland OpenCV иногда падает при выборе ROI. Варианты:

1. Запуск с **X11-бэкендом для Qt**:  
   `QT_QPA_PLATFORM=xcb uv run python main.py --video "videos/видео 1.mp4"`
2. Временно без ручного ROI (вся картинка — зона):  
   `uv run python main.py --video "videos/видео 1.mp4" --skip-roi`

## Какое видео и какой столик

Ролики в `videos/` **не входят в репозиторий** (слишком тяжёлые для Git): положите свои MP4 в эту папку локально (каталог создан через `videos/.gitkeep`).

**Столик задаётся вручную:** на первом кадре открывается окно **`cv2.selectROI`** — выделите прямоугольник вокруг одного чётко видимого столика и подтвердите (Enter). Для отчёта зафиксируйте координаты из **`outputs/report.txt`** (строка `ROI столика (x, y, w, h): ...`).

*Пример формулировки для отчёта:* «Использовалось видео `videos/видео 1.mp4`; ROI выделен вокруг центрального столика в зоне обслуживания (см. координаты в `outputs/report.txt`).»

## Логика детекции событий

1. На каждом кадре запускается **YOLOv8n**, оставляются только детекты класса **person** (COCO id `0`).
2. Для каждого бокса считается **доля площади пересечения** с прямоугольником ROI столика; если она не ниже порога (`cleaning_detection.geometry.person_in_table_zone`, по умолчанию **8%**), человек считается находящимся у этого столика.
3. **Состояние кадра:** «занято», если есть хотя бы один такой человек; иначе «пусто».
4. **События (на переходах):**
   - первый кадр фиксирует начальное `empty` или `occupied`;
   - **пусто → занято:** `approach` и `occupied` (подход после периода без людей в зоне);
   - **занято → пусто:** `empty`.
5. **Метрика:** для каждого события `empty` берётся время до **следующего** `approach`; по всем таким интервалам считается **среднее** (секунды).

Гостей и персонал по униформе не разделяем — любой человек в ROI учитывается одинаково.

## Результат для выбранного видео

После запуска смотрите консоль и **`outputs/report.txt`**: там число кадров, ROI, список метрик и строка вида:

`Среднее время между уходом (пустой стол) и подходом следующего: X.XXX с`

**Заполните этот раздел у себя после прогона** — значение зависит от выбранного ролика и ROI.

Также создаётся **`outputs/output.mp4`** — то же видео с прямоугольником ROI: **зелёный** — пусто, **красный** — занято.

## Пример проблемного кадра (скриншот)

Сохраните кадр из плеера или кадр из `outputs/output.mp4` (например, где детекция «спорная») и положите рядом, например `docs/problem_frame.png`, и укажите путь в отчёте/документации курса. В репозитории скриншот не добавлялся — его нужно сделать локально.

## Соответствие ТЗ (кратко)

|------------|------------|
| Детекция людей (YOLO или фон) | YOLOv8n, класс person |

| Зависимости | `requirements.txt` и/или `uv sync` |

## Структура проекта

- `main.py` — точка входа и цикл по видео.
- `cleaning_detection/` — геометрия ROI, события, аналитика (удобно покрывать тестами).
- `tests/` — юнит-тесты без реального видео и без GUI.
- **`outputs/`** — по умолчанию сюда пишутся размеченное видео и текстовый отчёт (каталог создаётся автоматически).
- **`logs/`** — файлы логов Loguru (отдельно от артефактов пайплайна).
